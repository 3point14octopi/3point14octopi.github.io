The Idle Hand (2022)

The Idle Hand is an isometric puzzle game created in a custom 3D C++ engine. Development lasted about eight months and was done in a team of 5.

In The Idle Hand, players control a floating hand sent by Satan to wreck havoc on an unsuspecting household.
For this project, I primarily developed several underlying systems, most noteably an interpolation system, an interaction system, and an animation system.

Interpolation System

The interpolation system was responsible for the locomotion within our game. It allowed the user to define a list of "keyframe" transformations, and the system would smoothly interpolate through each one. Position, Rotation, and Scale could be adjusted with this system, and speed could be independantly set for all three. A series of transformations (an "animation") could be set to happen just once, or continuously loop. Animations were written in a text file with a simple syntax, and saved to a folder within the game. When the interpolation behaviour was added to an object, the user would specify which script to use, and set it to be active or inactive.

Interaction System

The interaction system was very important for our game, as it was responsible for delivering feedback when the player interacted with objects in the house. When the player interacted with a valid object, several things would happen:
-the player would gain a new skin for their hand
-a screen displaying a picture of what the hand was doing would appear
-a sound would play
Several feedback actions would also occur. These feedback actions could include:
-an object's transformation changing
-an object's texture changing
-an object's mesh changing
-an object's animation changing or stopping
-an object's locomotion behaviour changing
There was no limit to how many of these could be performed upon any given interaction, and any object in the game could be affected by these actions. Since it would be unreasonable to hardcode each individual interaction, a system was devised where objects would receive an "interactable" behaviour, and would pass in one or more "feedback" type objects. Feedback objects had overloads to account for the above actions, and allowed for easy setup on the objects to be interacted with. Upon being triggered by the player, the interaction behaviour would perform each of the various feedbacks provided.

Animation System

Our group was provided with an OBJ renderer, and some VAO + VBO boiler plate code. To implement animation, I needed to go to the root of the VAO + VBO system and add a custom option for morph animation. I also needed to write a morph shader, and a custom behaviour that would update the meshes that needed to be interpolated between on a user-designated time interval.
